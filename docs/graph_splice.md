# graph! スプライス構文と extend の統一 (Fudaba #13 / #16)

2026-07-18 オーケストレータ決定 (ユーザーから続行指示「それを進めよう」)。

## 1. スプライス構文 (#13)

graph! リテラルの項として `..式` を許す。式は実行時コレクションで、リテラルの
静的な項と 1 つのリテラル内で合成される:

```rust
let staff: Vec<(String, Person)> = load_staff();          // 実行時データ
let deps: Vec<(String, DependsOn)> = load_deps();

let g = graphite::graph!(Org {
    root = Person { name: "CEO".into() },   // 静的な項 (識別子キー)
    ..staff,                                  // スプライス (実行時キー)
    ..deps,
    root -[..]-> ..                           // (静的エッジも従来どおり共存)
});
```

- 記法は Rust の struct update `..rest` の借用。項の先頭が `..` ならスプライス
- 式の型: `IntoIterator<Item = (K, T)>`、K: Into<String>、T はノード型または
  辺種別 (どちらかはマクロでなく **rustc の型推論が決める** — v3 以来の方針)
- 脱糖: `__graphite_b.extend(式);` (§2 の統一 extend)。戻り値の Id 列は捨てる
  (スプライスの要素は名前を持たない — 名前は静的な項だけの概念)
- 実行順: 静的ノードの let 束縛列 (従来どおり先頭) → 以降、静的エッジと
  スプライスを**記述順**に実行。検証は freeze 時なので意味論は順序に依存しない
  が、(0..*) 系の挿入順保証には記述順がそのまま現れる
- **キーの二重体系の説明** (本仕様の約束): 静的な項のキーは識別子 (書いた名前が
  そのままキー文字列)、スプライスのキーは実行時の String。どちらも「キー = 名前」
  (#4 の決定) の同じ意味論で、違いは名前の出所 (ソースコード上 vs データ上) だけ

## 2. extend の統一 (#13 の帰結、破壊的変更)

スプライスの脱糖先は「ノードか辺かを型推論で選ぶ」必要があるため、builder の
一括挿入を**単一の総称 `extend`** に統一する:

```rust
b.extend(node_pairs);   // (K, Person)   → ノードとして登録
b.extend(edge_pairs);   // (K, DependsOn) → 辺として登録
```

- 実装: ノード用 trait ({Schema}Node) と辺用 trait ({Schema}Edge) を単一の
  登録 trait に橋渡しする (共通 supertrait か blanket 橋 — 生成コード最小の
  方で。判断はコメントに)
- **`extend_nodes` / `extend_edges` は廃止** (直前に入れたばかりだが、統一
  extend と併存させると「やり方が三つ」になる。既定方針どおり痕跡なしで削除、
  org-analyzer の dataset.rs は `extend` へ書き換え)
- 要素単位の `insert` (ノード) / `add` (辺) は維持 (名前が挙動を語る主要 API)。
  → 検討: これも統一できるか? しない。insert/add はキーと値を個別に渡す
  「1 件の文」であり、名前の違い (ノードを挿す/辺を張る) に意味がある。
  extend は「データを流し込む」1 語で十分

## 3. 複数形構文の廃止 (#16)

`node 型名(複数形);` を廃止する。v4 でストレージ名は内部専用になり、明示する
意義が消えた。既定方針どおり検出・移行診断なし。使用箇所 (テスト
explicit_plural_field.rs 等) は削除または通常形へ移行。内部ストレージ名は
従来どおり素朴な複数形化で生成してよい (ユーザーに見えないため品質不問)。

## 4. テスト・実証

- スプライス: ノードのみ / 辺のみ / 静的項との混在 / 空コレクション /
  (0..*) の挿入順 (静的→スプライス→静的 の記述順が保存されること) /
  trybuild (スプライス式が (K,T) でない場合の rustc エラーが妥当なスパンに出ること)
- extend 統一: org-analyzer dataset.rs の書き換えが実証
- IDE: スプライス式内のトークンは素通しなので既存機構で解決するはず (実装後に
  オーケストレータが実測)
