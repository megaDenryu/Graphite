§1問題意識 — 二つのゴール

出発点の要求はシンプルだった: 独自構文 (graph_schema! / graph!) を書いたとき、VSCode が普通の Rust と同じように反応してほしい。参照ジャンプ、型追跡、rename。

しかし IDE の不具合を潰していくと、根っこがほぼ毎回同じ場所 — 「構文やマクロが、読み手に見えない何かを裏で発明している」こと — に行き着いた。隠れた生成型、文字列連結で合成されたメソッド名、識別子に見えて識別子でないトークン。IDE はこれらを解決できないし、人間も誤読する。だからこの記録の本当の主題は「構文が嘘をつかない言語にする」であり、IDE 対応はその検知器だった。

§2足跡 I — 計測、誤診、そして真因

まず測った

07-14

rust-analyzer の definition provider を直接叩き、F12 の正確な着地点まで確認する計測手法を確立。意外にもスパン保存は既に優秀で、本当の欠陥は 3 つだった: (1) graph! のノードキーが文字列に脱糖され LSP に不可視、(2) examples が未解析、(3) 編集途中にパースが壊れると生成型が全滅。いずれもスパンの付け替えでは直らない構造問題。let 束縛化・linkedProjects・宣言単位のエラー回復で解決した。

誤診の記録 — rename と「ケース変換の壁」

07-14

F2でbossをboss_edgeに変えてみたらなんか一括でいろいろ変わりました。…boss_edge_pair()をctrl+clickするとなぜか関数の定義ではなく、boss_edgeに飛びますが、これは適切？

F2 rename は派生名までカスケードしたのに、マクロが裏で生成していた BossAttrs だけが取り残された。最初の診断は「スパン継承漏れ」で、修正しても再発。真因は「rust-analyzer の rename は派生名に元トークンが原文ママ含まれる場合しか部分置換できない — PascalCase 変換を跨げない」という壁。名前を文字列変換で合成する設計そのものが病巣だと確定した。

§3足跡 II — 構文の書き換え (v2 / v3)

v2 — 「TypeScriptかよ」: 隠れた nominal 型の廃止

07-14

確かに{since:i32}て名前ないの？何こいつ？てなってました。typescriptかよ！…無名型みたいなやつは意味不明だし、コードも構造化を邪魔してただの横着なので犯罪だと思っています。

ノード型も辺の積み荷型もユーザーがマクロの外で普通の struct として宣言し、schema は参照するだけに。マクロは「グラフの機械」だけを生成することになった。旧構文は互換層も移行診断もなしの完全廃止 —

旧構文のにおいを一切残さないで欲しい。言語をクリーンに保って

v3 — 「= のほうが自然」: 束縛としての正直さ

07-15

グラフインスタンスの中のノードインスタンスなのだから、alice: Person { … } ではなく、alice = Person { … }のほうが自然な気がする。

リテラルは 名前 = 式 に。マクロは型名を読むのをやめ、trait + 総称メソッド経由で rustc の型推論に委譲する設計転換。副産物としてハンドシェイクマクロが全廃でき、同一ファイル制約と二段マクロ展開の IDE 制約が同時に消えた。

「テンプレート文字列」批判 — 導出名 API の全廃

07-16

テンプレート文字列みたいなもんじゃないですか？確かに参照ジャンプできますが、これを読まないと無理ですし、覚えることが多すぎて複雑すぎると思います。

ラベル 1 個から boss / try_boss / boss_id / boss_ids / boss_pairs … を文字列連結で合成する設計は、ケース変換の壁と同族の病 — むしろ親玉だった。ビュー方式 (固定語彙) を経て、最終的に v4 の型名前空間アクセスへ収束する。

§4足跡 III — 存在論の選び直し (v4)

「boss の型が BossEdge ではない」— 構文の嘘の摘発

07-16

「boss の型が BossEdge」ではないの部分を何とか解決したいですね。BossEdgeはbossの何なのか？が宣言的に書けるようになってないといけないと思います。

そして辺そのものへの問いが続く:

edge depends_on: Service -> Service は…DependsOnであるべきだし、vscodeでは緑で認識されるべきでしょう。…db -[depends_on]-> config よりも depends_on = DependsOn(db -> config) の方がいいかも？

v4 — 辺の第一級化: 多重グラフを基盤に宣言する

07-16〜17

存在論を選び直した: 辺種別は nominal 型 (生成される本物のタプル struct)、辺 1 本はキーを持つ第一級のインスタンス。ノードとの非対称が消滅。さらに「多重度は辺型の自己宣言ではなく、ノードの接続本数の制約では」というユーザーの直観が決め手になり、多重度注釈 (0..1) は where 制約に置き換わった。アクセスは型名前空間 (Boss::of(&g, ..)) になり、g.メソッド とビュー 6 型は全廃。

v4.1 — 端点役割名と無向辺: 矢印の意味を言語内に

07-17

矢印というのはそれがどういう意味を持ってるのかで向きの解釈も何もかも変わる。したがって言語として記述するときはそこを明確にするようにしないといけないし、できるような記法になっていないといけない。

この原理から 2 つのプリミティブが生まれた。端点役割名 (subordinate: Employee) -> (superior: Employee) — 向きの意味がコメントでなく言語内に固定され、入次数制約も書けるように。無向辺 Person -- Person — 対称な関係を矢印で書くのは嘘だった、という欠落プリミティブの補充 (ユーザー提起)。同時にモデリングガイド (グラフで書くべきもの vs 構造体で書くべきもの、二項関係の 4 分類、積み荷 = 間柄の属性) が確定した。

§5足跡 IV — 矢印の意味論と flow!

flow! — データフロー矢印の輸入

07-17〜18

Vertexには 引数 -[関数]-> 帰り値 の構文もあったはずですが、それは今どうなってますか？…むしろいれないといけないんじゃないか。脱糖するだけだから問題ない気がするんだけど。

過去に「原則 6 違反」と説明していたのは誤用で、訂正した。パイプ |> が中間値を隠すのに対し、矢印は中間値に名前と型を与える — という Vertex の設計を、fan-out / fan-in 込みで輸入 (「計算グラフを見据えてました」というスコープ確定を受けて)。

graphite::flow! {
    input -[parse]-> parsed,
    parsed -[validate]-> valid,          // fan-out
    parsed -[stats]-> report,
    (valid, report) -[merge]-> out,      // fan-in
};

脱糖は let y = (f)(x); の列のみ (消去可能)。「データの辺は宣言 (graph!)、関数の辺は実行 (flow!)」という統一 reading が構文レベルで成立。さらにその先の「計算グラフを実行時の値として持つ」レイヤは ComputeGraph (遅延評価 + 差分再計算のランタイムエンジン) として実装された。

§6足跡 V — キーとは何か (v4.2)

需要駆動の先送り、という誤り

07-18

Id 型のユーザー宣言化を「実需要が出たら再訪」と先送りしようとしたところ:

今てなに？意味不明な問ですね。言語ですよこれ？

言語は永続する成果物であり、問いは「いつ要るか」ではなく「意味論として何が正しいか」。枠組みごと問い直した。

v4.2 — キーは個体の名前

07-18

二つの首尾一貫した意味論 — (a) キーはグラフ内の名前 / (b) キーは個体の名前 — を比較し、ユーザーが (b) を選択。グラフとは、名前を持つ個体たちについての主張の集合である。導かれた規則:

「型を宣言した者が、その Id も宣言する」

Person はユーザーが宣言する → PersonId もユーザーが宣言する (その 1 行は「この型は名前で識別される個体である」という意味のある宣言)。Boss は schema が作る → BossId は生成のまま。複数の schema が同じ PersonId を共有できるのは、衝突ではなく当然の帰結になった — 組織図で得たキーで承認フローを引ける。

§7完成した世界観

// 意味を書く型は全部ユーザーの宣言。Id の宣言 = 「個体になれる」の宣言
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct PersonId(pub String);
pub struct Person { pub name: String }
pub struct BossEdge { pub since: i32 }

graphite::graph_schema! {
    schema Org {
        node Person;
        node Team;

        edge BelongsTo = Person -> Team                          where each Person: 1;
        edge Boss      = (subordinate: Person) -[BossEdge]-> (superior: Person)
                                                                  where each subordinate: 0..1;
        edge Friends   = Person -- Person                        where unique pair;
    }
}

let g = graphite::graph!(Org {
    alice = Person { name: "Alice".into() },       // 名前 = 定義
    bob   = Person { name: "Bob".into() },
    b1    = Boss(bob -[BossEdge { since: 2021 }]-> alice),
    f1    = Friends(alice -- bob),
    ..load_more_people(),                           // 実行時データの合成
})?;

let boss = Boss::of(&g, &bob_id);                  // アクセスは型名前空間
let subs = Boss::sources_of(&g, &alice_id);        // 逆引き
規則意味
名前 = 定義名前が要る定義は schema もリテラルも全部 名前 = 値/型式。リテラルの名前は常にキーの束縛
矢印の中は積み荷だけ-[X]-> の X は schema では積み荷の型、リテラルでは積み荷の値、flow! では関数
where は制約多重度・一意性は辺の自己宣言ではなく、宣言される制約 (each 役割: 1 / unique pair)
キーは個体の名前グラフ = 個体宇宙への主張の集合。型を宣言した者が Id も宣言する

§8外延 — この過程で得た設計原理

構文は嘘をつかない

名前: 型 の顔をしたものは本当にその型宣言でなければならない。「〜っぽく見えるが実は違う」は、それ自体が欠陥。

導出名は病気

文字列変換で名前を合成すると IDE と人間の両方が壊れる。名前はユーザートークン原文ママか、固定語彙か。例外は Rust 自身の慣例が要求する変換 (enum バリアント等) のみ。

借り物の読み方は本物から借りる

新しい構文は Rust プログラマが既に持つ読解 (関数型の矢印、タプル struct のコンストラクタ、let 束縛、struct update の ..) に乗せる。本物に実在しない形の発明は新しい嘘。

マクロは推論をエミュレートせず rustc に投げる

マクロが型名をテキストで読むほど、見えない結合が増える。trait + ジェネリクスに委譲すれば、型検査・推論・エラーの全部が本物になる。

検証は実測で。誤診は記録する

rust-analyzer の挙動は definition provider を直接叩いて確かめる。誤診 → 修正 → 再発 → 真因、の往復こそが「ケース変換の壁」のような法則の発見に繋がる。

互換の痕跡を残さない

未配布の言語に移行診断・互換層は書かない。旧構文の検出コードは「旧構文の知識」をコードベースに残すこと自体が汚染。

言語の意味論を需要駆動で先送りしない

「実需要が出たら」は アプリケーションコードの論理。言語の問いは「いつ要るか」ではなく「何が正しいか」。保留できるのは「正しさが未決」のときだけ。

完了した札に種を埋めない

完了列は誰も読み返さない。将来の作業は、閉じる前に「出自 + 着手条件」つきの未着手札へ切り出す。
