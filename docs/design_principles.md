# Graphite 設計原則

**「Graphite の設計は Rust 的な精神の遵守。型については strict」** — これが
Graphite 全体を貫く最上位方針である。フェーズ5後半でこの方針を以下の 6 原則に
具体化した。以後、新機能の設計判断に迷ったらまずこの 6 原則に照らして考える。

## 系譜: なぜこの原則群が必要になったか

Graphite は `../Bullet/docs/rust_graph_extension_sketch.md` /
`../Bullet/docs/graph_design_sketches.md` (Vertex 側の設計検討文書) から派生した
プロジェクトである。`graph_design_sketches.md` の「決定まとめ」(同ノード同一性 =
ユーザーキー、可変性 = クロージャスコープ builder→凍結、多重度 = freeze で一括検査
等) は、いずれも「グラフ固有の新発明を持ち込まず、既存の Rust の型システムと
所有権にそのまま乗せる」という一貫した態度の産物であり、Graphite の
`graph_schema!`/`graph!` はその態度をマクロ生成コードとして実装したものである。

同文書はさらに「既存言語 (Rust) の拡張として作れないか」という問いを検討し、
TypeScript:JavaScript の類推を分解している。TS が成立した条件の一つは
「追加が全部消去可能だった (`interface` も型注釈も消せば JS のまま)」ことであり、
唯一の例外である TS の `enum` (消去できず実行時コードを生成する拡張) は
「TS チームが後年まで悔やむ設計」として名指しされている。ここから引ける教訓は
「ホスト言語への拡張は消去可能な範囲では上手くいき、意味論を持ち込んだ瞬間に
牙を剥く」ということであり、Graphite の `graphite-macros` が生成するコードにも
そのままあてはまる: **マクロは Rust の意味論に何も足してはならず、手書きの
Rust コードに脱糖できる範囲にとどまるべきである**。原則5・6 はこの教訓を
直接の起源とする。

## 6 原則

### 原則1: 型の strictness — stringly-typed API 禁止

キーや識別子は生の `String`/`&str` で運ばず、newtype 型 (`EmployeeId(String)`
のような) や専用の enum バリアントで運ぶ。「文字列の中身で意味を判別する」設計
(stringly-typed) は、コンパイラの型検査を素通りしてしまい、原則1 が守ろうとする
Rust 的な型安全性そのものを失う。

**Graphite での具体例**: フェーズ5以前の `{Schema}Violation::MultiplicityViolation`
は違反キーを `source: String` (`Debug` 表現への `format!` 出力) として運んでいた。
「エッジごとに始点ノード型が異なりうるので型を1つに固定できない」という理由での
妥協だったが、これは stringly-typed API そのものだった。フェーズ5 (項目k) で
エッジ単位に型付きバリアントを生成する形
(`BelongsToMultiplicity { source: EmployeeId, count: usize }` /
`BelongsToUnknownSource { key: EmployeeId }` など) へ置き換え、`String` を
経由せずコンパイル時に型が効くようにした。

### 原則2: パニックは契約違反のみ

パニックを許すのは「呼び出し規約 (contract) の違反」の場合に限る。入力検証の
欠如や通常のドメインエラーでパニックさせてはならない。パニックする関数には
必ず非パニック版の `try_` 関数を対で用意し、パニックする関数の doc コメントに
`# Panics` 節を書いて条件を明記する。

**Graphite での具体例**: 多重度 `(1)` のアクセサ `{label}(&SrcId) -> &T` は
「このスキーマが発行したキー以外を渡す」という契約違反時のみパニックし、
`# Panics` 節にその旨を明記している。対になる `try_{label}(&SrcId) -> Option<&T>`
が非パニック版として必ず生成される (`Vec` の `v[i]`/`v.get(i)` と同じ関係)。
フェーズ5で追加した ID 版アクセサ `{label}_id`/`try_{label}_id` も同じ対称性を
踏襲する。

### 原則3: std 命名規約準拠

`try_` プレフィックス、`_with_key` のようなサフィックス、イテレータを返す
メソッドは `impl Iterator<Item = ...>` を返す、といった標準ライブラリの命名・
設計規約にマクロ生成コードも合わせる。利用者が Rust の直感で API を予測できる
ことを優先する。

**Graphite での具体例**: `filter_nodes`/`map_nodes` に対する「キーも渡す版」は
`filter_nodes_with_key`/`map_nodes_with_key` という命名にした
(`HashMap::retain` 等、std のキー付き版命名に倣う)。`{node_snake}_ids()` /
`{label}_pairs()` はすべて `impl Iterator<Item = ...>` を返し、`Vec` を
即座に確保しない。

### 原則4: 借用検査と戦わないAPI

「借用チェッカを回避するための定型儀式」を利用者に強いる API を避け、
所有権ベースのヘルパー (clone を明示的に要求する / 値を返す等) を用意する。
借用の絡む定型変換は、利用者が毎回自力で書くのではなくライブラリ側に用意する。

**Graphite での具体例**: `{label}_pairs()` が返す `&K` を汎用
`Graph<N, E, K>` へ射影する際に借用エラーを踏みやすかった問題
(`docs/phase5_candidates.md` 項目b) に対し、`Graph::from_edges(nodes, edges)`
という所有権ベースのヘルパーを追加し、`.cloned()` で値を渡せば済む形にした。

### 原則5: ゼロコスト志向

マクロが生成するコードは、人間が手で書いたコードと同形でなければならない。
実行時リフレクションや動的ディスパッチ (`dyn Trait` 越しの汎用ディスパッチ、
文字列によるフィールド名解決等) をマクロ生成コードに持ち込まない。

**Graphite での具体例**: `crates/graphite/tests/orgchart_handwritten.rs`
(フェーズ2の手書きテンプレート) と `graph_schema!` の生成コードは、
`cargo expand` で比較すると同形になるように作られている。違反 enum も
実行時に文字列でエッジ種別を判別するのではなく、コンパイル時に確定した
専用バリアント (原則1) として展開されるため、`match` はすべて静的に解決される。

### 原則6: 消去可能な拡張のみ

マクロが追加する構文・API は、脱糖後の実行意味論を一切変えてはならない。
「マクロを手で展開したら普通の Rust コードになる」という性質 (消去可能性) を
壊す拡張は持ち込まない。

この原則の起源は Vertex 側の検討 (`../Bullet/docs/graph_design_sketches.md`
「TS の enum は反例ではなく警告だった」節) にある。TypeScript が JavaScript の
拡張として成立した条件の一つは「追加が全部消去可能だった」ことで、唯一の
例外である TS の `enum` (消去できず実行時コードを生成する) は TS チーム自身が
後年悔やんだ設計とされる。教訓は「ホスト言語への拡張は消去可能な範囲では
上手くいき、意味論を持ち込んだ瞬間に牙を剥く」ということであり、Graphite の
`graphite-macros` にもそのままあてはまる。

**Graphite での具体例**: `graph_schema!`/`graph!` はどちらも「脱糖すれば
手書きの struct 定義・builder メソッド呼び出し列になる」マクロであり、
生成コードは実行時に一切マクロ固有のランタイム (リフレクション用メタデータ、
動的ディスパッチテーブル等) を必要としない。`graph!` が生成する
`SchemaName::create(|b| { ... })` の呼び出し列は、利用者が手で書く builder
呼び出しと文字通り同じコードである。

## 適用時の判断フロー

新しい API・マクロ生成コードを設計する際は、以下の順で 6 原則に照らす:

1. キーや識別子を `String`/`&str` のまま運んでいないか (原則1)
2. パニックする関数に `try_` 版と `# Panics` があるか (原則2)
3. 命名が std の慣習から外れていないか (原則3)
4. 利用者に借用の定型儀式を強いていないか (原則4)
5. 生成コードは手書きコードと同形か、実行時リフレクション/動的ディスパッチを
   持ち込んでいないか (原則5)
6. マクロの追加が脱糖 (消去) で説明できるか、実行意味論を変えていないか (原則6)

## 既知の残課題

6 原則を新規適用する過程で見つかった、既存コードに残る違反・グレーゾーンは
`docs/phase5_candidates.md` および各コミットメッセージに記録する。次フェーズ
着手時はまずそこを確認すること。
