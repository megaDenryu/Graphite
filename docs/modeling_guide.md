# モデリングガイド — グラフで書くべきもの、構造体で書くべきもの

2026-07-16〜17 の議論 (Fudaba #8) の結論。発端はユーザーの問い:

> 矢印の始点終点は edge.start とかでアクセスできるので良いと思っていたが、
> 状況によっては変数名 (役割名) がついていた方がいい気がしてきた。しかしそれは
> 構造体で書くべきものかもしれない。グラフで書くべきものと構造体で書くべきものの
> 違いをもっと認識しないといけない。

および設計原理:

> 矢印というのはそれがどういう意味を持っているかで向きの解釈も何もかも変わる。
> 言語として記述するときはそこを明確にできる記法になっていないといけない。

## 1. 大原則: グラフの要素にしてよいのは「同一性 + 接続性」を持つものだけ

- **キー (同一性) で参照され、他の要素と繋がる**ものだけをノード/辺にする。
- 共有も参照もされない値 (`name: String`、`price: u32`) はただのフィールド。
  `Department` がノードなのは複数の社員から共有参照されるから。`name` を
  ノードにしないのは誰もそれを参照しないから。
- 迷ったら問う: 「この値に 2 箇所以上から矢印が刺さるか？」— 刺さらないなら
  フィールド。
- 実装上のコロラリー (v4.2、`docs/node_id_v4_2.md`): ある型の隣に
  `{型名}Id` を宣言することは、単なる配管作業ではなく「この型は名前で
  識別される個体であり、ノードになれる」という宣言そのものである。

## 2. 二項関係の書き分け (4 分類)

| 関係の性質 | 記法 | 例 |
|---|---|---|
| 向きが意味を持ち、from/to で自然に読める | `edge Kind = A -> B` | DependsOn, BelongsTo, 遷移 |
| 向きはあるが、端点を役割名で読みたい | `edge Kind = (役割: A) -> (役割: B)` | Boss (subordinate → superior), Trade (buyer → seller) |
| 対称 (役割の区別が存在しない) | `edge Kind = A -- A` | Friends, 相互接続, 隣接 |
| 3 項以上の関係 / 関係にさらに関係が付く | **ノード昇格** (§4) | 取引 (buyer, seller, item), 承認つき任命 |

判定の核: **端点の呼び名が矢印の向きから導出できるか**。
- 導出できる (from = 依存元, to = 依存先) → 1 行目
- 導出できないが 2 項で非対称 (buyer/seller — どちらを from にするかが規約で
  決まる) → 2 行目 (役割名が規約を言語内に固定する)
- そもそも向きが無い → 3 行目
- 端点が 3 つ以上、または関係そのものを指差したい → 4 行目

## 3. ペイロード (積み荷) の判定 — 「間柄の属性」だけを積む

積み荷に書いてよいのは、**from にも to にも属さず「from と to の間柄」に属する値**だけ。

判定質問: 「この値は from のものか？ to のものか？ 間柄のものか？」
- `BossEdge { since }` — 就任年は部下のものでも上司のものでもなく上司**関係**の属性 ✓
- `Role { name: "lead" }` (Assigned) — 役割は人でもプロジェクトでもなく間柄の属性 ✓
- 反例: 積み荷で**端点を呼び分け**始めたら (`{ side: "buyer" }` 等)、それは間柄の
  属性ではなく端点の役割 → 役割名つき有向 (§2 の 2 行目) か昇格 (§4) に移す

積み荷なしの辺が多数派で正常 (examples の belongs_to / depends_on / feeds /
遷移辺はみな積み荷なし)。積み荷は「関係に日付・重み・理由が付くとき」の少数派機能。

## 4. 昇格 (関係 → ノード) の手順

「取引」を辺として書こうとして詰まったら (3 項ある / 取引自体を参照したい):

```rust
// 昇格前 (書けない): edge Trade = Person -[?]-> Person   ← item はどこへ? 向きは?

// 昇格後: 「取引」という関係そのものをノードにし、
//         端点の役割は昇格後ノードから生える辺の種別名になる
node Trade;
edge Buyer  = (trade: Trade) -> (buyer: Person);
edge Seller = (trade: Trade) -> (seller: Person);
edge Item   = (trade: Trade) -> (item: Product);
```

- 昇格するもの = **辺として書こうとしていた関係そのもの** (表現形式が
  辺 1 本 → ノード 1 個 + 辺数本に変わる)
- v4 で辺はキー付きの第一級要素なので、昇格は機械的 (辺キーがそのまま
  昇格後ノードのキーになる)
- 昇格後の辺の向きは「関係ノード → 参加者」で統一すると読みやすい
  (役割名が意味を運ぶので、向き自体は規約でよい — それを役割名が言語内に固定する)

## 5. 実例: 同種の辺の間の役割差は「辺種別を分ける」

reactive-cells の減算セル `Sub` は入力 2 本の**区別** (被減数/減数) が要る。
これは端点役割名では書けない (2 本の辺が同じ種別・同じ端点役割になるため)。
「同種の辺の間の役割差」が要る場合は、**役割を辺種別に昇格**する:

```rust
// 誤: edge Feeds = Cell -> Cell を 2 本張り、どちらが左かを別の場所で管理 (二重管理)
// 正: 役割が辺の種類の名前になる
edge Lhs = (operand: Cell) -> (consumer: Cell);
edge Rhs = (operand: Cell) -> (consumer: Cell);
```

役割は、値 (積み荷) でも位置 (引数順) でもなく、**名前 (nominal な辺種別)** で
書く — 本プロジェクトの一貫した原則の適用。

判定基準の具体化 (reactive-cells への適用で得られた知見): **演算が可換なら
入力の区別は不要なので 1 種の辺で束ねてよい** (Sum/Mul は `Feeds` のまま)。
**非可換な演算だけが役割差を必要とする** (Sub のみ `Lhs`/`Rhs` に分割)。
種別分割は「役割差が要るときだけ」— 予防的に全部を分割しない。

## 6. まとめ (チェックリスト)

1. それはキーで参照され、繋がるか？ → No ならフィールド
2. 関係は 2 項か？ → No ならノード昇格
3. 端点の呼び名は向きから出るか？ → No なら役割名、向き自体が無いなら `--`
4. 積み荷は「間柄の属性」か？ → 端点の役割が混ざっていたら移す
5. 同種の辺同士に役割差が要るか？ → 辺種別を分ける
