# flow! — データフロー矢印 (中間値に名前と型を与えるパイプ) (Fudaba #11)

2026-07-18 オーケストレータ決定 (ユーザーから全面委任。スコープは 2026-07-17 の
議論でユーザー確定済み: 「計算グラフを見据える」→ fan-out/fan-in を含む)。

## 出自

Vertex の `x -[f]-> y` (../Bullet/docs/dataflow_design.md)。動機は `x |> f |> g` の
可読性問題 — パイプは中間値を隠し、**矢印は中間値に名前と型を与える**。
`x -[f]-> y -[g]-> z` ≡ `let y = f(x); let z = g(y);`

## 構文

```rust
graphite::flow! {
    input -[parse]-> parsed,
    parsed -[validate]-> valid,          // fan-out: parsed を
    parsed -[stats]-> report,            //   2 本の矢印に流す
    (valid, report) -[merge]-> out,      // fan-in: タプル始点
};
println!("{}", out.summary);             // 束縛は flow! の後で普通に見える
```

- **文位置のマクロ**。項はカンマ区切りの「矢印文」:
  `始点 -[関数式]-> 束縛名`
- 始点: 任意の式、または `(式, 式, ..)` (fan-in。関数は多引数で呼ばれる)。
  2 項目以降は前で束縛した名前をそのまま書くのが典型 (ただの変数参照)
- `-[f]->` の f: 任意の式 (関数パス・メソッドへの参照・クロージャ変数)。
  脱糖は `let 束縛名 = (f)(始点..);` (括弧で包み、クロージャ式等も安全に呼ぶ)
- 束縛名: 新しい ident。**graph! と同じく let 束縛として外に漏れる**
  (文位置 proc マクロ + call-site スパン — G1 で実証済みの機構)
- チェーン形 `x -[f]-> y -[g]-> z` (1 項の中で連鎖) も許す
  (≡ `x -[f]-> y, y -[g]-> z` の糖衣。Vertex の原形)

## 意味論

- **即時実行の純粋な脱糖** (消去可能な拡張)。項の記述順に let 文を並べるだけ。
  先行未定義の名前を使えば rustc の普通のエラー
- fan-out の所有権 (move 済み値の再利用) は rustc の標準エラーに任せる —
  正直な脱糖なので Rust の所有権規則がそのまま見える。参照で流したい場合は
  `&parsed -[f]-> y` のように始点式側で借用する (始点は任意の式なので追加機構
  不要)
- `?`/`.await` 等の後置は v1 では矢印文に混ぜない (始点式・関数式の中では自由)。
  必要になったら再訪
- **具象化された計算グラフ** (グラフを実行時の値として持ち、遅延実行・差分
  再計算) は本マクロの範囲外の別レイヤ (reactive-cells の一般化)。flow! と同じ
  「ノード = 値、辺 = 関数」の形を共有する — 将来 flow! の記述をエンジンに
  載せ替える道を閉じない

## 実装ノート

- graphite-macros に `flow` proc マクロを追加、graphite から re-export
  (graph!/graph_schema! と同じ構成)
- スパン: 束縛名・関数式・始点式は全てユーザートークン素通し (G3)。
  f への定義ジャンプ・束縛名の hover 型表示が機能すること (実装後に実測)
- エラー回復: 項単位 (G4 と同じ方針)。壊れた項は compile_error! 蓄積 + 残りを生成
- 重複束縛名の診断 (graph! のキー重複と同じ親切さ: 最初の宣言位置併記)
- テスト: 直線・チェーン形・fan-out・fan-in・クロージャ関数式・束縛の後方可視・
  trybuild (重複束縛名・壊れた項の回復)
- 教材: hello-graph に「§5 flow! — 関数の辺」を追加 (graph! との対比:
  データの辺は宣言、関数の辺は実行。決定3 の統一 reading を 1 段落で)
