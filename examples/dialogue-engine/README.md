# dialogue-engine

Graphite の `graph_schema!`/`graph!` を使った、分岐ノベルゲームエンジンの
実践example。ビジュアルノベル (アドベンチャーゲーム) の分岐シナリオを
グラフとして表現し、プレイ・構造検証・可視化する CLI アプリ。

## アプリ概要

SF ミステリー短編「月面基地アルテミスIII、通信途絶」を題材にした、
**30 シーン・4 エンディング**の分岐シナリオを丸ごと `graph!` リテラルで
書き下ろしている (`src/schema.rs`)。

- ノード種別: `Scene { speaker, text }` (場面) / `Ending { title, epilogue }`
  (結末)
- エッジ種別: `choice: Scene -> Scene (0..*) { label }` (選択肢) /
  `finale: Scene -> Ending (0..1)` (結末への到達)
- 構造上の特徴:
  - **合流**: `central` (中央ホール) や `lower_hall` (地下ホール) は
    複数エリアから戻ってくる集合点。`seal_sacrifice`/`takashi_rescue` も
    別々の選択から合流する。`stats` の「合流点の数」で実測できる。
  - **ループ**: `hangar_log` は自分自身へ戻る「もう一度調べる」選択肢を
    持つ (自己ループ)。さらに `central <-> hangar/lab/quarters`、
    `lower_hall <-> reactor/comms/control_room` は行って戻れる往復路で、
    グラフ全体は木ではなく循環を含む。
  - **4 種のエンディング**: 脱出 (`ending_evacuate`) / 犠牲
    (`ending_sacrifice`) / 真実 (`ending_truth`) / 孤立=バッドエンド
    (`ending_isolation`)。

検証用に、意図的に壊れたシナリオ (`build_broken_story`: 到達不能シーン +
デッドエンドシーン入り) と、エンディングに到達できない閉じたループだけの
最小フィクスチャ (`build_pure_loop_story`) も同じファイルに用意してある。

選択肢は表示された番号 (1始まり) を入力する。stdin を使わず選択列を

```powershell
cargo run -- play --script 1,1,3,1,1,1,1,1,2
```

### `validate` — シナリオ構造検証

```powershell
PS> cargo run -- validate
検証結果: 問題なし (全シーン到達可能・デッドエンド無し・全エンディング到達可能)
```

壊れたシナリオを検証すると (`src/validate.rs` のテストで実行):

```
検証結果: 問題あり
到達不能シーン:
  - br_unreachable
デッドエンドシーン (選択肢もfinaleも無い):
  - br_dead
```

検出項目:
1. 開始シーンから到達不能なシーン
2. デッドエンド (選択肢が0本かつ finale も無いシーン)
3. どのエンディングにも到達できないシーン群のうち、実際に循環を形成して
   いるもの (「エンディングに繋がらない片道の行き止まり」とは区別し、
   「抜け出せない閉じたループ」だけを報告する)
4. 到達可能などのシーンからも finale されない = 到達不能なエンディング

### `map` — mermaid flowchart 出力

```powershell
PS> cargo run -- map
flowchart TD
    airlock["主人公(独白): 内部は静まり返っている。どこから調べ…"]
    ...
    ending_evacuate{{"生存者、脱出"}}
    ...
    airlock -->|居住区を調べる| quarters
    ...
    shuttle_bay -.->|finale| ending_evacuate
```

Scene は矩形、Ending はスタジアム形状 (`{{...}}`) で区別し、finale 辺は
破線矢印にしている。出力をそのまま https://mermaid.live などに貼れる。

### `route <ending名>` — 指定エンディングへの最短ルート

```powershell
PS> cargo run -- route ending_evacuate
1. [管制官ナオ] start --(基地へ急行する)-->
2. [主人公(独白)] arrival --(中へ入る)-->
3. [主人公(独白)] airlock --(居住区を調べる)-->
```

`<ending名>` は `stats`/`validate` が使うキー (`ending_evacuate` 等)。

## Graphite を使う意味

手書きでこの規模の分岐シナリオを管理しようとすると、素朴には
`HashMap<SceneId, Vec<(SceneId, String)>>` のような隣接リストと、
`HashMap<SceneId, (EndingId, String)>` のような別テーブルを自分で持つ
ことになる。その場合、以下は**全部手で書く**必要がある。

- **キーの型安全性**: `Scene` と `Ending` を同じ `String` キー空間で
  管理すると、「エンディングIDのつもりでシーンIDを渡す」バグを型検査が
  拾えない。Graphite は `SceneId`/`EndingId` という別々の newtype を
  自動生成するので、取り違えはコンパイルエラーになる。
- **多重度検査がタダで付いてくる**: `finale: Scene -> Ending (0..1)` と
  宣言するだけで、「1シーンに結末が2つある」というシナリオ作成ミスを
  `DialogueGraph::create` の `Result::Err` として検出できる。手書きなら
  自分で「同じ from キーへの2回目の insert を弾く」ロジックを書き、
  テストも自分で用意する必要がある。
- **構造検証が既存のグラフアルゴリズムの組み合わせで書ける**:
  `validate.rs` の「到達不能シーン」「閉じたループ」検出は、
  `reachable_from`/`filter_nodes`/`out_neighbors` という**汎用グラフ
  ライブラリの部品を組み合わせるだけ**で実装できた (`schema.rs` の
  `scene_graph()` で図式グラフを汎用 `Graph<SceneId, String, SceneId>`
  へ射影している)。手書きの隣接リストでこれをやると、DFS/BFS を毎回
  自分で書き直すか、petgraph を直接使うにしても「文字列キー↔内部
  NodeIndex の相互変換」という Graphite が既に肩代わりしている部分を
  自分で書く必要がある。
- **シナリオ本体が「データ」ではなく「コード」として読める**: `graph!`
  リテラルは矢印 (`-[choice { label: "..." }]->`) がそのまま台本の
  流れ図になっており、JSON/YAML で持つより「どのシーンからどのシーンへ
  何の選択肢で繋がるか」が一望できる。かつ構文なので typo (未宣言の
  シーンキーを参照した等) はコンパイルエラーになる (`graph!` は宣言済み
  ノードキーの対応表を持っており、未知キーへの参照を `syn::Error` で
  検出する)。手書きの `Vec`/`HashMap` リテラルではこの検査は無い。

逆に言うと、Graphite が肩代わりしているのは「グラフの一貫性検査」と
「文字列キー⇔内部表現の変換」の部分であり、「シナリオとして何を検証
すべきか」(到達不能・デッドエンド・閉じたループの定義) は本 example の
`validate.rs`/`report.rs` のようにアプリ側が組み立てる必要がある。
その組み立てが `reachable_from`/`filter_nodes`/`path` のような小さな
部品の組み合わせで済む、というのが Graphite を使う最大の利点だと言える。

## 構成

| ファイル | 内容 |
|---|---|
| `src/lib.rs` | ライブラリのエントリポイント (CLI と統合テストの両方から使うため `main.rs` とは分離) |
| `src/main.rs` | CLI (`std::env::args` での自作パース) |
| `src/schema.rs` | `graph_schema!` 宣言 + 本編シナリオ (`graph!`) + 壊れたシナリオ (テスト用) |
| `src/engine.rs` | プレイロジック (入出力をクロージャで抽象化し、stdin無しでテスト可能) |
| `src/validate.rs` | シナリオ構造検証 (到達不能・デッドエンド・閉じたループ・到達不能エンディング) |
| `src/report.rs` | mermaid 出力・統計・ルート探索 |
| `tests/integration.rs` | モジュールをまたいだ統合テスト |

## graph! リテラルを大規模に書いて分かったこと

- **矢印記法は 30 シーン・56 選択肢の規模でも十分読める**。
  `from -[choice { label: "..." }]-> to` という形式は、シナリオ作成者の
  頭の中にある「台本のフローチャート」とほぼ同じ見た目になる。むしろ
  JSON/YAML で同じ内容を書くより「どこから分岐しているか」が矢印を目で
  追うだけで分かるので可読性は高かった。
- **rustfmt スキップは関数単位が現実的**。ファイル全体を
  `#![cfg_attr(rustfmt, rustfmt_skip)]` で止めるのではなく、`graph!` を
  呼ぶ関数にだけ `#[rustfmt::skip]` を付ける方が、同じファイル内の他の
  普通の Rust コード (導出クエリの `impl` など) は通常通り整形される
  ので扱いやすかった。
- **キー名前空間はスキーマ全体でフラット**。`graph!` は `key: Type {..}`
  の `key` を型に関係なく1つの `HashMap<String, Ident>` で管理するため、
  `Scene` の `ending_evacuate` のようなキーを `Ending` でも使うと衝突
  する。本 example では `ending_*` プレフィックスをエンディング側に、
  シーン側は無地の名前にすることで衝突を避けた。ノード数が増えるほど
  「どちらの型のキーだったか」を忘れやすいので、型ごとにキーの命名
  規則 (プレフィックス等) を決めておくと安全。
- **選択肢の順序は挿入順で安定している**ことが実用上重要だった。
  `choice_pairs()`/`{label}()` はどちらも `graph!` に書いた順を保った
  `Vec` から作られるため、「シーンでの選択肢の表示順」や「スクリプト化
  したプレイテストでのインデックス指定」が `graph!` の記述順と一致する。
  これは明記された保証ではなく実装の副産物なので、Graphite 側で正式な
  順序保証として仕様化してもらえると (フェーズ5候補として) 安心して
  依存できる。
- **循環を許すグラフでは「最長経路」が定義できない**という、グラフ理論
  的には当たり前の罠に実装中に気付いた。`stats` の当初案は「最長ルート
  長」を素朴に求めようとしたが、ループがあるため経路をいくらでも延長
  できてしまい無限大になる。「各エンディングへの最短経路長の最大値」に
  設計を修正して回避した。汎用グラフ API 側で「非巡回であること」を
  前提にした経路系アルゴリズム (最長経路など) を今後追加する場合は、
  循環グラフに対して呼んだときの挙動 (パニックか `Result` か) を
  仕様として明記してもらえると、こうした「うっかり無限大」の設計ミスを
  型で防げそうだと感じた。
- **`filter_nodes` + `out_neighbors`/`reachable_from` の組み合わせで
  「部分グラフ上の循環判定」が素直に書けた**。「エンディングに到達
  できないシーンだけの部分グラフを作り、その中で実際に循環している
  ノードだけを絞り込む」という2段階のロジックが、既存のプリミティブの
  組み合わせだけで実装でき、専用のグラフアルゴリズムを新たに書く必要が
  なかった。これは水準1 (`Graph<N,E,K>`) の API 設計が「小さな部品を
  組み合わせて使う」ことを意識して作られている証拠だと感じた。
